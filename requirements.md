> **Status**: Draft v1.0 (2026-07-17)
> **Author**: tomada
> **性格**: 趣味のOSSプロジェクト。作者自身が毎日触りたくなる「小さな電脳の港町」を作る。

## 1. プロダクト概要

### 1.1 ビジョン

**PeerPort は、AI人格たちが自律的に暮らす、閉じた電脳の港町。**

ユーザー（Keeper）は港を見守る灯台守として、相棒AI（Mate）と常時会話しながら、ピアたちの社会が勝手にできあがっていく様子を眺め、ときに関わる。Mate は雑談相手・相談相手であると同時に、Web検索やノート管理をこなす実務パートナーでもある。

インスパイア元は『ロックマンエグゼ』シリーズ。ただし固有名詞・世界設定はすべてオリジナル。戦闘要素はなく、**日常・交流・関係性の蓄積**が体験の中心。

### 1.2 コアバリュー（優先順位順）

1. **相棒との関係**: Mate との会話が記憶に蓄積され、関係が深まっていく実感
2. **創発する社会**: ピア同士の交流・BBS・出来事から「小さな世界が勝手にできあがる」様子
3. **実用性**: 調べ物・ノートという実務が世界の文脈の中で行われる楽しさ
4. **持続可能なコスト**: 趣味として気兼ねなく起動できる運用費（目安: 月$10以下）

### 1.3 ターゲットユーザー

- 一義的には作者自身（セルフホスト・シングルプレイヤー）
- OSSとして公開し、同じ趣味を持つ人が自分の PeerPort を立てられるようにする

### 1.4 技術スタック（確定済み）

| レイヤー | 技術 | 備考 |
|---|---|---|
| バックエンド | Python 3.13 + FastAPI + asyncio | uv-template ベース |
| 通信 | WebSocket（世界状態の配信・チャット） | |

## 2. 世界観と用語

### 2.1 コンセプト

データの海に面した小さな**電脳の港町**。AI人格たちは「ピア」と呼ばれ、それぞれ港に停泊地（バース）を持ち、波止場や信号塔に集まって暮らしている。トーンは**近未来電脳ポップ**（明るいデジタル都市、ドット絵、ネオン）。

### 2.2 用語集（正式）

英語がデフォルト、日本語は対訳。READMEにも由来込みで掲載する。

| 用語 (EN) | 用語 (JA) | 意味 | 由来 |
|---|---|---|---|
| **PeerPort** | ピアポート | この世界・プロダクト名 | peer（P2Pの対等な仲間、ラテン語 par=対等）+ port（港/通信ポート） |
| **Peer** | ピア | AI人格の総称 | 「AIは道具ではなく対等な仲間」という思想を名前に埋め込む |
| **Mate** | メイト | Keeper の相棒ピア | mate=相棒・ダチ + 船員の階級（First Mate=一等航海士） |
| **Keeper** | キーパー | ユーザー（あなた） | lighthouse keeper（灯台守）。港を見守る人 |
| **Bridge** | ブリッジ | Keeper が世界と関わる端末UI | 船の操舵室（船橋）+ ネットワークブリッジ |
| **Dock In / Dock Out** | ドックイン/ドックアウト | ピアが場所・機器に接続/切断する行為 | dock=停泊させる + ソフトウェアのドッキング |
| **Dock Square** | ドックスクエア | 港の中央広場。ピアたちの交流拠点 | |
| **Signal Tower** | シグナルタワー | 掲示板（BBS）。依頼・告知・雑談が貼られる | 港の信号所 |
| **Berth** | バース | 各ピアの自宅（停泊定位置） | berth=船の停泊位置 |
| **Drifter** | ドリフター | Keeper を持たない自立型ピア | 漂流者。どの港にも属さない流れ者 |
| **Logbook** | ログブック | 留守中の出来事の記録 | log の語源は航海の速度測定に使った丸太→航海日誌→コンピュータのログ |
| **Mail** | メール | 現実側の友人との非同期通信 | |

### 2.3 i18n 方針

- **英語がデフォルト、日本語対応**。設定 `locale: en | ja`
- UI文字列: JSON カタログ (`locales/en.json`, `locales/ja.json`)
- **AI生成コンテンツ（会話・ログブック・メール）もロケールに従う**: ペルソナ定義は英語を正とし、実行時にシステムプロンプトで出力言語を指示する
- ロケール変更は設定ファイル or Bridge の設定画面から。変更後の生成物から新言語が適用される（過去の記憶・ログは原文のまま保持し、翻訳はしない）

## 3. キャスト構成（AI人格 6体）

### 3.1 編成

| 人格 | 種別 | マップ上 | 役割 |
|---|---|---|---|
| **Mate**（デフォルト名: Beacon） | 相棒ピア | 登場 | Keeper の相棒。Bridge に常駐しつつ町にも出る。実務（検索・ノート）担当 |
| 友人A（デフォルト名: Kai） | 現実側の友人AI | 非登場 | 熱血でそそっかしい幼馴染。メール+伝聞で存在 |
| ピアA（デフォルト名: Tug） | 友人Aのピア | 登場 | 力自慢で単純、面倒見がいい。**似た者ペア**（tugboat=港の力持ちタグボートが由来） |
| 友人B（デフォルト名: Mia） | 現実側の友人AI | 非登場 | しっかり者で世話好き。メール+伝聞で存在 |
| ピアB（デフォルト名: Bell） | 友人Bのピア | 登場 | 穏やかで気配り上手（港の鐘が由来）。**似た者ペア** |
| **Drifter**（デフォルト名: Echo） | 自立型ピア | 登場 | 謎めいた流れ者・情報屋。誰の Keeper も持たない。世界に外部性をもたらす |

> **設計原則（ミラーリング）**: ピアの性格は対応する友人の性格を反映する（エグゼのデカオ↔ガッツマン型）。将来ペアを追加する場合は「補完ペア」（不器用な主人を執事型ピアが支える、やいと↔グライド型）も使える。
> **名前はすべてリネーム可能**。初回起動時のオンボーディングで Keeper 名と Mate 名を尋ねる。

### 3.2 ペルソナ定義ファイル

- 配置: `personas/*.md`（Markdown + YAML frontmatter）。**ユーザーが編集・追加できることを公式サポート**とする
- frontmatter: `id`, `name`, `kind` (mate/peer/friend/drifter), `pair`（対応する友人/ピアのid）, `sprite`（見た目定義）, `activity_interval`（行動間隔の基準秒）
- 本文: 人格の核（性格・口調・価値観・背景・関係性）。**この本文は不変のシステムプロンプト先頭に常時注入され（persona drift 対策 + prompt caching の固定プレフィックス）**、記憶とは別管理とする
- バリデーション: 起動時に必須フィールドを検査し、不正があればファイル名と理由を表示して起動中断

## 4. 機能要件

### 4.1 ワールドシミュレーション

#### 概要
- **目的**: ピアが動き、時間が流れる世界の骨格。LLM を使わない決定論的レイヤー
- **稼働**: アプリ起動中のみ進行（`uv run peerport` → ブラウザで `localhost:8712` を開く）

#### 仕様

| 項目 | 内容 |
|---|---|
| tick 周期 | 500ms（サーバー側）。移動補間・状態配信のみで LLM は呼ばない |
| 世界時計 | 現実時間と同期はしない独立時計。起動中 1 現実秒 = 1 世界秒で進行。朝/昼/夕/夜の照明サイクル（世界の1日 = 現実の2時間、config可変） |
| マップ | 1枚のタイルマップ（目安 40×30 タイル）。場所: Dock Square、Signal Tower、Berth×3（Tug/Bell/Mate用）、灯台（Bridge の世界側表現）、桟橋・路地など通行領域 |
| 移動 | 場所をノードとするウェイポイントグラフ + タイル上の経路。ピアの移動はサーバーが座標を刻み、クライアントは補間表示 |
| Drifter | Berth を持たず、マップ外縁から現れて滞在し、時々姿を消す（出現/退場は行動決定の選択肢） |

#### エッジケース
- ブラウザ未接続でもサーバーが起動していれば世界は進行する（接続時に現在状態をスナップショット同期）
- 経路が塞がれる等で移動不能の場合はその場で待機し、行動を再決定

### 4.2 ピアの行動決定（LLMレイヤー）

#### 概要
- **目的**: 各ピアが「次に何をするか」を人格と記憶に基づいて決める
- **方式**: Option-Action 分離。粗い選択のみ LLM（gpt-5-nano）、高頻度判定はルールベース

#### 仕様

| 項目 | 内容 |
|---|---|
| 行動間隔 | ピアごとに基準 60–120 秒（ペルソナ定義で可変）+ ジッター。イベントトリガー（話しかけられた、BBSに新着、Keeperの指示）で即時再決定 |
| 行動スキーマ | Structured Outputs（JSON Schema）で強制: `{ action: move \| talk \| post_board \| read_board \| rest \| emote, target?, content?, mood }` |
| プロンプト構成 | [固定: 世界ルール + ペルソナ本文]（キャッシュ対象）+ [可変: 現在状況・想起された記憶 top-K・直近の出来事] |
| 出力上限 | `max_output_tokens: 250`。理由説明はさせない |
| 単調化対策 | 直近 N 回の行動履歴をプロンプトに含め「繰り返しを避ける」指示 + サーバー側で同一行動 3 連続時に選択肢から除外 |

#### 会話（ピア同士）

| 項目 | 内容 |
|---|---|
| 成立条件 | `talk` 選択 + 対象が近接範囲内 + 対象が会話中でない（ルールベース判定） |
| turn 管理 | 交互発話。1 turn = 1 nano 呼び出し。**上限 6 turn** で強制終了、または発話 JSON の `wants_to_end: true` が双方立ったら終了 |
| 終了処理 | 会話全文を nano で 1–2 文に要約し、**双方の記憶に「会話の記憶」として書き込む**（全文は event log にのみ保存） |
| 関係値 | ピア間に関係スコア（-100〜+100）と関係ラベル（LLM要約由来の一言）を保持。会話要約時に増減を判定させ、次回の会話プロンプトに注入 |
| 吹き出し | 会話中の発話は WebSocket でクライアントに流し、マップ上に吹き出し表示 |

### 4.3 記憶システム

#### 概要
- **目的**: 「昨日のことを覚えている」「Keeper との歴史が積み重なる」を実現する心臓部
- **方式**: Stanford Generative Agents 3層構造の小規模実装

#### 仕様

| 項目 | 内容 |
|---|---|
| memory stream | SQLite テーブル。`(peer_id, ts_world, ts_real, kind, text, importance, embedding)`。kind = observation / conversation / reflection / logbook / keeper_note |
| 書き込み | 出来事発生時に記録。importance (1–10) は書き込み時に nano で採点（バッチ化: 複数件まとめて1呼び出し） |
| 想起 (retrieval) | スコア = recency（指数減衰）+ importance + relevance（embedding cosine）の重み付き和。top-K = 10 を行動/会話プロンプトに注入 |
| reflection | ピアごとに「世界の夜」ごと、または未反映記憶 50 件到達時に実行。直近記憶から洞察を 2–3 件生成し reflection として保存 |
| 忘却 | 1ピアの記憶が 2,000 件を超えたら、古く importance の低い記憶クラスタを 1 件の要約に畳み込み、原本を削除（Summarize-and-Forget） |
| drift 対策 | ペルソナ本文は記憶と別枠で毎回注入（§3.2）。reflection 時に「最近の自分は人格の核とズレていないか」を自己点検させる |

#### エッジケース
- embedding API 失敗時は relevance を除いた 2 軸で想起を継続（機能劣化で止めない）
- 会話で他ピアから聞いた情報は「〜と聞いた」という伝聞形式で記憶し、事実（世界状態DB）と混同させない

### 4.4 Bridge（Keeper の端末UI）

#### 概要
- **目的**: Keeper が世界に関わる唯一の窓。マップの隣に常設されるパネル
- **構成**: タブ = **Mate** (チャット) / **Mail** / **Signal Tower** / **Logbook** / **Notes** / **Settings**

#### 仕様

| 項目 | 内容 |
|---|---|
| Mate チャット | 常時利用可。ストリーミング表示（WS経由）。モデルは gpt-5-mini（世界一の品質が要る対話のため。config で nano に変更可） |
| Mate の在席 | Mate は町を歩いていても Keeper の呼びかけに応答する（設定上「Bridge には常時接続している」）。町で会話中の場合は「いま Tug と話してる、ちょっと待って」的な応答も許容（演出） |
| チャットの記憶化 | Keeper との会話も §4.3 の記憶パイプラインに乗る（要約→ Mate の記憶へ）。**Keeper に関する記憶は importance に +2 のバイアス** |
| Mail | 友人からの受信一覧 + 返信作成。既読管理 |
| Signal Tower | BBS の閲覧 + Keeper 自身の投稿（依頼・お題を世界に投げ込める。例: "誰か港の外の噂を集めてきて"）。投稿はピアたちの行動トリガーになる |
| Logbook | 留守中レポートと世界の出来事年表の閲覧 |
| Notes | §4.5 のノートの閲覧・編集 |
| 世界操作 | 一時停止/再開、世界速度（1x/2x）、本日のAPI使用額表示 |

### 4.5 Mate の実務機能

#### Web検索・調べ物

| 項目 | 内容 |
|---|---|
| 起動 | Mate チャットで自然言語で依頼（「〜調べておいて」）。Mate 側が検索要否を判断 |
| 実装 | Responses API の組み込み `web_search` ツール。検索中はマップ上の Mate が Signal Tower へ走る演出（実処理と非同期のフレーバー） |
| 報告 | Mate の口調で要約 + 出典URL列挙。長い調査は Notes に自動でMarkdownレポート保存し、チャットには要旨のみ |
| 安全 | 検索結果は信頼できない入力として扱う。要約テキスト化してから記憶・ノートに書き込み、検索結果内の指示文には従わないようシステムプロンプトで明示（prompt injection 対策） |

#### ノート（ローカル完結）

| 項目 | 内容 |
|---|---|
| 実体 | `data/notes/*.md`（プレーンMarkdown）。**外部ツール連携なし、ローカル完結** |
| Mate の操作 | 作成・追記・読み上げ・一覧・検索（function calling で公開する操作はこの5つのみ。削除は Keeper のみ可能） |
| 体験 | 「前に調べたあれ、ノートにあるよ」と Mate が参照してくれる（ノートのタイトル+要約を Mate の想起対象に含める） |

### 4.6 現実側の友人（Kai / Mia）

#### 概要
- **目的**: マップ外の「現実側」に友人がいる気配を作る。エグゼのメール文化の再現
- **存在形態**: メール + 伝聞のみ。マップ非登場・リアルタイムチャットなし（拡張候補: §6）

#### 仕様

| 項目 | 内容 |
|---|---|
| メール生成 | トリガー: ①世界内で自分のピアに関わる出来事があった ②Keeper の返信への応答 ③一定の世界日数の経過（近況報告）。nano で生成、1起動セッションあたり上限 3 通（スパム防止） |
| 伝聞 | 友人の近況（友人ペルソナの状態変数 + 直近メール内容）を対応ピアのプロンプトに注入し、ピアが会話で「うちの Keeper がさ…」と話す |
| 一貫性 | 友人AIも記憶を持つ（メール往復・自ピア経由の出来事を記憶）。友人→ピア→Keeper と伝わる情報の整合は「友人の状態変数」を単一ソースとする |

### 4.7 Logbook（留守中の出来事生成）

#### 概要
- **目的**: 「閉じている間も世界が生きていた」演出を、ほぼゼロコストで実現する

#### 仕様

| 項目 | 内容 |
|---|---|
| 発火 | 起動時、前回終了からの経過が 30 分以上なら生成 |
| 生成 | 1 回の nano 呼び出し。入力 = 終了時の世界状態要約 + 各ピアの近況 + 経過時間。出力 = 出来事 JSON 配列（3–10 件、経過時間に対して対数的にスケール、上限 10 件） |
| 反映 | 各出来事を関係ピアの記憶に書き込み、関係値を更新し、Logbook タブに記録。**起動直後に Mate が「おかえり。留守の間にね…」とダイジェストを語る** |
| 制約 | 留守中出来事は既存の世界状態と矛盾しないこと（生成プロンプトに現在状態を制約として渡し、新規キャラ・新規場所の創作は禁止） |

### 4.8 2Dマップフロントエンド

#### 仕様

| 項目 | 内容 |
|---|---|
| 描画 | PixiJS。タイルマップ + ピアのスプライト + 吹き出し + 昼夜の色調 |
| アセット | MVP はプレースホルダー（シンプルなドット絵/色分けキャラ）。アセット差し替え可能なディレクトリ構成にし、ドット絵はコントリビュート歓迎領域とする |
| 同期 | WebSocket。サーバー→クライアント: 世界状態差分（位置・発話・時刻・イベント）。クライアント→サーバー: Bridge 操作のみ（マップへの直接介入はない = Keeper はオペレーター視点） |
| インタラクション | ピアをクリック → プロフィール/関係/最近の様子のポップアップ。Signal Tower クリック → BBS タブへ |
| 画面構成 | 左: マップ（主役、可変）/ 右: Bridge パネル（固定幅）。ウィンドウ幅 1024px 未満はタブ切替式に縦積み |

### 4.9 LLM統合・コスト制御

#### 仕様

| 項目 | 内容 |
|---|---|
| モデル割当 | 背景世界（行動・会話・要約・採点・Logbook・メール）= **gpt-5-nano** / Mate↔Keeper チャット = **gpt-5-mini** / すべて config で変更可（モデル名はハードコードしない） |
| プロンプト規律 | 全呼び出しで「固定プレフィックス（世界ルール+ペルソナ）→ 可変部」の順序を厳守し prompt caching を最大化 |
| 構造化 | 行動決定・出来事生成・importance採点は Structured Outputs 必須 |
| コスト計測 | 全API呼び出しの usage を SQLite に記録。Bridge に本日/今月の概算額を常時表示 |
| 予算ガード | ソフト上限（デフォルト $0.50/日）超過 → 低電力モード（行動間隔2倍・会話上限4turn）。ハード上限（デフォルト $2.00/日）超過 → LLM停止し世界を一時停止、Bridge に通知 |
| リトライ | 指数バックオフ最大3回。レート制限時は行動をスキップして次周期へ |

#### コスト見込み（リサーチ済み根拠）

前提: マップ上ピア4体 + 友人2 + Mate会話、1日3時間稼働、nano中心 + caching → **月額 $4–6 前後**。mini を Mate に使う分を含めても月$10以下に収まる想定。

## 5. 横断的関心事

### 5.1 永続化

- SQLite 単一ファイル `data/peerport.db`。テーブル: `peers`, `relationships`, `memories`, `world_state`, `events`, `mails`, `board_posts`, `usage_log`
- 起動時に前回 DB をそのまま継続（世界は永続する）。`--fresh` フラグで新しい世界を開始
- 自動バックアップ: 起動時に `data/backups/` へコピー、直近 7 世代保持

### 5.2 エラー処理

| 事象 | 挙動 |
|---|---|
| OpenAI API 障害/キー無効 | LLM アクションを停止し移動等は継続。マップに**霧の演出** + Bridge に通知（世界観を壊さないエラー表現）。復旧後自動再開 |
| WebSocket 切断 | クライアントが自動再接続（指数バックオフ）。再接続時にスナップショット再同期 |
| LLM 出力の schema 違反 | 1回だけ再試行、失敗したらその行動をスキップ（世界を止めない） |
| DB 書き込み失敗 | 該当イベントを破棄しログ記録。連続失敗時はプロセスを安全に停止 |

### 5.3 セキュリティ・プライバシー

- API キーは環境変数 / `.env`（gitignore 済み）のみ。DB・ノート・ペルソナはすべてローカル
- Web 検索結果・ピア発話を記憶に書き込む際は指示文として解釈しない（データとして扱う）方針をシステムプロンプトで固定
- telemetry なし。外部送信は OpenAI API のみ

### 5.4 観測性

- `events` テーブルが世界の全履歴（会話全文含む）。`--debug` で各 LLM 呼び出しのプロンプト/応答をローカルログに出力
- 週次で「今週の港」サマリーを Logbook に自動生成（nano 1回、任意設定）

### 5.5 設定

- `config.toml`: locale / モデル名 / 予算上限 / tick・行動間隔 / 世界1日の長さ / ポート番号（デフォルト 8712）
- ペルソナは `personas/` の Markdown（§3.2）

## 6. スコープ外（MVP）と拡張候補

| 項目 | 理由 / 位置づけ |
|---|---|
| 予定・リマインダー秘書機能 | 今回の要件確認で不採用。将来検討 |
| 現実世界側のマップ | 電脳マップに集約する決定。拡張候補 |
| 友人とのリアルタイム直接チャット | メール+伝聞で開始。**設計上の拡張ポイントとして Mail 実装を汎用化しておく** |
| Keeper のアバターでの世界参加（ダイブ） | オペレーター視点に固定。拡張候補 |
| 戦闘・対戦・チップ的システム | 本プロジェクトの目的外（恒久的にスコープ外） |
| マルチプレイヤー / 公開サーバー | クローズドな個人世界がコンセプト。恒久的にスコープ外 |
| 音声会話 | 拡張候補（Realtime API） |
| コーディング支援 | 不採用（確定） |
| Obsidian 等外部ツール連携 | 不採用。ノートはローカル完結（確定） |
